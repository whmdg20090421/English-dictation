# 自动化打包与发布检测规范

当被要求发布 APK 或 Tag 时，必须使用浏览器实时监测远端构建流水线，并确认最终产品真实可用后才能结束任务。必须遵循以下流程：

**第一步**：使用 `git commit` 和 `git push` 等将本地代码推送到远端仓库触发构建。

**第二步**：必须实时使用浏览器监测其编译状态。在编译过程中，必须使用 `sleep 60` 暂停60秒，然后再刷新浏览器网页并截取快照。因为编译进程即使失败了也可能在未刷新的页面上显示为“正在编译中”，不会自动同步失败信息。必须循环执行“暂停60秒 -> 刷新网页 -> 截取快照”的动作，若编译报错失败，必须在本地修改报错代码并重新触发编译，直到编译成功为止。

**第三步**：只有当编译成功之后，必须通过浏览器进入安装包发布页（Releases/Tags）检查是否已发布对应的 `.apk` 安装包，并截取该页面的快照（截图） 之后才能宣告任务完成并结束运行。

**注**：严禁在编译失败或没有截图快照的情况下结束任务。浏览器是必须使用的监测与确认工具。

# Git 提交信息规范

在使用 Git 进行代码提交（git commit）时，应遵循 Conventional Commits 规范，但尽量使用中文作为提交信息的描述主体（description）。例如：`feat(ui): 增加中文界面支持` 或 `fix(auth): 修复登录失败的问题`，以便于团队内部更直观地理解变更内容。

# 版本号与更新规则

当进行 Bug 修复、代码修改或功能调整时，**除非用户明确下令要求更新版本号**，否则**绝对不要**自动修改 `pubspec.yaml` 中的版本号（version）和构建号（versionCode / `+` 后面的数字）。
只有当用户明确喊你“升级版本”或“版本加一”时，你才被允许同时增加版本号（如 `1.0.7` -> `1.0.8`）和 versionCode（如 `+3` -> `+4`）。请时刻记住当前的版本状态，并在用户指令明确时严格执行双重增加（version 和 versionCode 都必须增加）。
