# 移动端终极安全架构：基于 Argon2id 与 TEE 的鉴权与端到端加密报告

## 1. 执行摘要 (Executive Summary)
本报告针对应用目前的加密和鉴权逻辑进行了深度的安全审查，并根据最新需求设计了一套符合金融级和端到端加密（E2EE）标准的移动端安全架构。新架构彻底分离了“管理员鉴权”与“数据加密”的职责，引入了防暴力破解的 Argon2id 密钥派生算法，并结合了操作系统的可信执行环境（TEE）。

## 2. 需求分析与安全隐患 (Vulnerability Analysis)
### 2.1 当前隐患
- **职责混淆**：之前的设计或思路中，管理员密码和加密密钥的职责界限模糊。
- **缺乏防暴力破解机制**：直接使用明文密码进行加密或比对，若黑客获取本地或云端数据，可通过 GPU 高速字典攻击暴力破解。
- **密钥明文落地**：虽然上一个版本修复了 SharedPreferences 的问题，但依然需要更严密的防 Hook 设计。

### 2.2 最新安全目标
1. **管理员密码**仅用于身份验证和配置更改，绝不直接参与业务数据的加密。
2. **密钥加密密码（MEK Password）**专门用于推导主加密密钥，且该密码在内存中“用完即焚”，不长期留存。
3. **防暴力破解**：必须引入高消耗的 KDF（密钥派生函数）并配合盐值（Salt），增加单次破解的计算成本；在 UI 层面增加错误重试锁定机制。

## 3. 架构重构方案 (Architectural Design)

### 3.1 密码学原语选择
- **密钥派生算法 (KDF)**：废弃简单的 SHA-256，全面采用 **Argon2id**。这是目前公认最抗 GPU/ASIC 暴力破解的算法，通过调节内存消耗（Memory Cost）和迭代次数（Time Cost），让暴力破解的时间成本呈指数级上升。
- **对称加密算法**：继续使用 **AES-256-GCM**。
- **随机数生成器**：所有 Salt 和 Nonce 必须使用操作系统的 **CSPRNG**（加密安全伪随机数生成器）。

### 3.2 鉴权与加密分离流程
#### A. 管理员鉴权机制 (Admin Authentication)
- **注册/修改时**：系统生成一个随机盐（Admin_Salt）。使用 Argon2id(管理员密码, Admin_Salt) 生成一个哈希值（Admin_Hash）。将 `Admin_Salt` 和 `Admin_Hash` 存入本地数据库及云端配置中。
- **登录鉴权时**：用户输入密码。系统提取 `Admin_Salt`，再次运行 Argon2id 算法，将结果与 `Admin_Hash` 比对。一致则通过，允许修改配置。
- **防暴力破解（UI与逻辑层）**：在本地引入连续错误锁定机制（例如：连续错 5 次，锁定 15 分钟，锁定时间随次数指数递增）。

#### B. 数据加密机制 (Data Encryption)
- **设置加密密钥密码时**：用户输入“密钥加密密码”。系统生成随机盐（MEK_Salt）。使用 Argon2id(密钥加密密码, MEK_Salt) 派生出真正的 256位主加密密钥（**MEK**）。
- **数据加密**：使用这个派生出来的 **MEK**，配合 AES-256-GCM 算法对 SQLite 数据库和云端 JSON 数据进行加密。
- **MEK 的存储与读取**：将 MEK_Salt 存储在本地及云端。如果用户开启了“生物识别免密登录”，则将派生出的 MEK 存入 TEE（`flutter_secure_storage`），日常同步时自动读取。如果用户极其注重安全，可选择不存入 TEE，每次同步云端必须手动输入“密钥加密密码”重新派生 MEK。

## 4. 实施细节与修复计划 (Implementation Plan)
1. **引入 `cryptography` 包**：替换原有的 `crypto` 包，以支持 Argon2id 算法。
2. **重构 `CryptoUtils`**：实现基于 Argon2id 的 Hash 生成、校验以及 MEK 派生功能。
3. **重构 `CloudSetupScreen`**：分离管理员密码的 Hash 存储与“密钥加密密码”的派生逻辑。
4. **增加防暴力破解模块**：在 `SharedPreferences` 中记录 `login_attempts` 和 `lockout_until` 时间戳，限制管理员密码的频繁尝试。
5. **升级 TEE 配置**：确保 `flutter_secure_storage` 在 Android 上强制使用 `EncryptedSharedPreferences`，在 iOS 上使用最高安全等级的 Keychain 配置。

## 5. 结论 (Conclusion)
该架构方案不仅满足了用户“鉴权与加密分离”、“防暴力破解”的核心需求，还通过 Argon2id 和 TEE 的结合，在保证端到端加密（E2EE）安全性的同时，兼顾了跨设备同步的便利性。该报告及对应的开发规范（Spec）已生成，可随时进入实施阶段。
